# Lighthouse/relay validation gates

Add the lighthouse/relay validation gates that run **before any remote API call**, so a misconfigured host fails fast and creates no record: lighthouse and relay are mutually exclusive; a lighthouse needs ≥1 static address; a lighthouse or relay needs a non-zero listen port.

## Public interface

```go
// internal/enroll
func validateRoles(cfg *config.Config) error  // called at the top of Enroll, before remote calls
```

Inputs: `cfg.IsLighthouse`, `cfg.IsRelay`, `cfg.StaticAddrs`, `cfg.ListenPort` (parsed by [CFG.types](dt-toqi.md)). Returns a clear, specific error per rule.

## Behaviors (TDD order)

1. **Both lighthouse and relay → fail** — mutually exclusive error.
2. **Lighthouse without static address → fail** — `IsLighthouse && len(StaticAddrs)==0` → error.
3. **Lighthouse without non-zero port → fail** — `IsLighthouse && ListenPort==0` → error.
4. **Relay without non-zero port → fail** — `IsRelay && ListenPort==0` → error.
5. **Valid configs pass** — lighthouse w/ addr+port; relay w/ port; plain host (neither) — all return nil.
6. **Gates run before remote calls** — in an `Enroll` test, an invalid role config makes zero API calls.

## Test strategy

Table-driven `(Config) -> error|nil` for `validateRoles`. Plus one `Enroll`-level test asserting the mock api client is untouched on invalid input.

## Acceptance

- All four failure rules fire with specific messages; valid combinations pass; no remote call on invalid input.
