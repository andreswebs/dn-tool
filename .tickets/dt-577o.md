# DECISION: run attempts shutdown-unenroll even when enroll never happened

Decide whether run.Lifecycle should skip the shutdown unenroll when the host was never enrolled (install/enroll failed early). Currently it always tries, logging a misleading error. Benign but noisy.

## Design

run.Lifecycle (internal/run) always attempts the shutdown unenroll after the foreground returns, even when the lifecycle aborted early — i.e. install or enroll failed and the host was therefore never enrolled. This is visible on any non-Linux dev host: `dn-tool run` fails at the install OS-gate, then still logs `unenroll failed during shutdown; host may still be enrolled` (with `DN_API_KEY is required to unenroll` when no key is set).

## Where

internal/run/run.go — runAndUnenroll() calls deps.Unenroll(...) unconditionally after Run(...) returns, regardless of whether Run failed at the install or enroll step (before the daemon ever started).

## Why it is currently benign

Unenroll of a never-enrolled host is idempotent and safe: with no local dnclient config it returns dnclient.ErrNotEnrolled (no remote DELETE, no orphan); the §2.5 invariant is untouched. The daemon/lifecycle error still takes exit-code precedence (the unenroll error is logged, not returned). So this is a noise / clarity issue, not a correctness bug.

## The decision

Should runAndUnenroll skip the shutdown unenroll when enroll never succeeded?

Options:
1. Leave as-is. Simple; "always unenroll on termination" is the stated container contract (dt-n5p5). Cost: a confusing ERROR log on every failed-install/enroll run.
2. Track whether enroll succeeded and only unenroll then. Removes the misleading log. Cost: extra state in the lifecycle; must define "succeeded" (enroll Result returned without error — including the already-enrolled no-op, which SHOULD still unenroll on a clean run).
3. Keep always-unenroll but downgrade the not-enrolled outcome to a debug/info log (treat ErrNotEnrolled during shutdown as an expected no-op, not an ERROR). Smallest change; keeps the contract, kills the noise.

Recommendation to evaluate: option 3 (quietest, preserves the contract) vs option 2 (most correct intent). Decide before the Layer 2 e2e harness lands, since the harness will assert on run's shutdown behavior.

## Scope note

Discovered while wiring the run command (dt-s5tr). Not changed there — out of scope, and internal/run is already implemented/tested (epic dt-cmn7). This ticket owns the decision + any change.

## Notes

**2026-06-07T13:45:53Z**

DECISION: option 2, approach B. Gate the shutdown unenroll on the daemon having been reached. Run() now wraps pre-daemon (install/enroll) failures with sentinel errBeforeDaemon; runAndUnenroll skips Unenroll when errors.Is(err, errBeforeDaemon) and surfaces the wrapped phase error (maps to CodeError). Chosen over option 3 because the headline dev-host symptom logs 'DN_API_KEY is required to unenroll' (errMissingAPIKey, checked before ReadHostID) — NOT ErrNotEnrolled — so downgrading ErrNotEnrolled would not have silenced it. The 'already enrolled' no-op enroll returns nil, so a clean run still unenrolls on shutdown; the §2.7 container contract is untouched. Impl: internal/run/run.go + 2 tests (TestRunAndUnenroll_{Install,Enroll}FailureSkipsUnenroll). make build green.

**2026-06-07T20:28:33Z**

Mechanism revisited by dt-4l0b (architecture-review deepening candidate 4). The DECISION here is unchanged — gate the shutdown unenroll on the daemon having been reached — but the errBeforeDaemon sentinel that carried it through the error chain is being replaced with an explicit Run return: Run() (daemonStarted bool, error), runAndUnenroll gates on !started. Rationale: a future pre-daemon step that forgets to wrap with errBeforeDaemon would silently re-enable the misleading unenroll; an explicit return can't be forgotten. Behavior-preserving.
